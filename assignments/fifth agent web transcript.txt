We recently changed some API routes, updated our dashboard layout CSS, and added a new notifications widget. Do not redesign the site. Only fix what is necessary.

Please investigate this as a complex web application bug. Use any external tools available to inspect runtime behavior, code, and rendering. Identify the reproduction steps, failure scope, root cause, proposed patch, verification checks, regression risk, and release recommendation.

Best-supported diagnosis: this is most likely a compound dashboard failure introduced by this morning’s deploy, with the new notifications widget as the primary JavaScript crash source and the recent dashboard CSS changes as a secondary layout regression.

That points to a likely sequence like this:

Dashboard mounts.

Notifications widget requests the old endpoint (/api/notification singular).

Request 404s after the API route rename.

Widget code still tries to do .map(...) on an undefined response field.

Separate CSS/mobile breakpoint changes likely hide or collapse the sidebar more aggressively on smaller screens, which is why mobile/tablet users report it more often.

The uncontrolled-to-controlled warning is probably real but not the crash. It is usually a state initialization issue and should be cleaned up, but it is unlikely to be the reason for the white screen.

Proposed Patch
{patches}

This should be a minimal, non-redesign patch.

Patch 1 — fix the notifications endpoint
In src/api/notifications.js or wherever the widget fetches notifications:

change the request from /api/notification to the current route, almost certainly /api/notifications

ensure the client checks response.ok before consuming JSON

Best-guess patch shape:

export async function getNotifications() {
  const res = await fetch('/api/notifications');

  if (!res.ok) {
    return [];
  }

  const data = await res.json();
  return Array.isArray(data) ? data : (Array.isArray(data?.notifications) ? data.notifications : []);
}
Patch 2 — make the widget render-safe
In NotificationsWidget.jsx:

initialize notifications state to [], not undefined or null

guard all .map(...) calls with an array fallback

fail closed: if fetch fails, show empty state instead of throwing

Best-guess patch shape:

const [notifications, setNotifications] = useState([]);
const [error, setError] = useState(null);

useEffect(() => {
  let active = true;

  getNotifications()
    .then((items) => {
      if (active) setNotifications(Array.isArray(items) ? items : []);
    })
    .catch(() => {
      if (active) {
        setError(true);
        setNotifications([]);
      }
    });

  return () => {
    active = false;
  };
}, []);
And in render:

{notifications.length === 0 ? (
  <div className="notifications-empty">No notifications</div>
) : (
  notifications.map((item) => /* existing UI */)
)}
Patch 3 — isolate widget failure from blanking the whole dashboard
In Dashboard.jsx:

do not let notifications be required for the dashboard shell to render

render sidebar/layout first

treat notifications as optional content

Best-guess approach:

ensure the dashboard does not derive core layout from widget data

wrap widget render in safe conditional logic

if your app already has an error boundary pattern, place the widget under it; if not, avoid adding new libraries and just harden the widget itself

Patch 4 — fix mobile/sidebar CSS regression
In dashboard.css:

inspect recent breakpoint changes around tablet/mobile widths

revert only the rule causing sidebar disappearance

likely candidates:

display: none

width: 0

overflow: hidden

collapsed grid/flex columns

absolute positioning off-screen

z-index overlap from the new widget

Minimal CSS intent:

preserve current design

keep the sidebar visible or intentionally toggleable exactly as before deployment

prevent the notifications widget from forcing parent overflow/collapse

A likely safe correction pattern would be:

/* concept only */
.dashboard-layout {
  display: flex; /* or existing grid */
  min-height: 100vh;
}

.sidebar {
  flex: 0 0 var(--existing-sidebar-width);
}

.dashboard-main {
  min-width: 0; /* important for flex/grid overflow containment */
}
And remove any recent mobile rule that unintentionally hides the sidebar unless that behavior was explicitly intended.

Patch 5 — clean the uncontrolled input warning
Wherever the input is defined, initialize controlled values safely:

const [value, setValue] = useState('');
instead of undefined / null.

Again: fix it, but do not treat it as the blocker.
